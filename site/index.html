Before you start

What runs where

OmnigentThe optional local web, mobile, and multi-harness surface.
TailscalePrivate access from your phone without exposing the server publicly.
HarnessThe durable scheduler, state machine, QA loop, and evidence store.
OpenCodeThe authenticated worker CLI that edits and tests the project.

Run commands on the machine containing the repository. You need Git, Bash, Node.js, jq, one supported worker CLI, and its model credentials. Omnigent and Tailscale are optional.

Step 1

Install OpenCode

OpenCode writes the code in the tested setup. Codex plans and performs independent checks. Authenticate both CLIs before continuing.

curl -fsSL https://opencode.ai/install | bash
source ~/.zshrc  # use ~/.bashrc when running Bash
opencode         # run /connect, choose a provider, then exit
Checkopencode --version

Also check: codex --version. Claude Code can be used instead when it is authenticated.

Step 2

Install the harness and optional Omnigent bundle

Run the installer normally. Keep harness checked, toggle omnigent, and confirm once.

curl -sSL https://raw.githubusercontent.com/vinicius91carvalho/harness-engineering/main/install.sh | sh
Checktest -f ~/.config/opencode/skills/harness-planner/SKILL.mdtest -f ~/.omnigent/agents/harness-engineering/config.yaml

Native-only alternative: add --cli opencode --no when you do not want Omnigent.

Step 3

Choose project role models

Save this small file as .harness/roles.json. It says: OpenCode codes; Codex checks.

{
  "coding": [{"harness": "opencode"}],
  "validation": [{"harness": "codex"}],
  "repairPlanning": [{"harness": "codex"}],
  "goalReview": [{"harness": "codex"}]
}

More choices: copy ~/.omnigent/agents/harness-engineering/roles.example.json and edit its ordered fallbacks.

Step 4

Connect your phone privately

Install Tailscale on the server and phone, sign both into the same tailnet, and proxy the local Omnigent server over private HTTPS.

OMNIGENT_WS_ALLOWED_ORIGINS=https://YOUR-MACHINE.ts.net \
OMNIGENT_ACCOUNTS_BASE_URL=https://YOUR-MACHINE.ts.net \
  omni server start
tailscale serve https / http://localhost:8000
omni host

Security: use tailscale serve, not Funnel, for tailnet-only access. The server machine must remain awake.

Open: https://YOUR-MACHINE.ts.net on the phone. Omnigent's mobile UI carries chat, files, status, and Input Requests.

Step 5

Plan with Codex

Start Omnigent from the Git root and ask Codex to create the project specification. Describe behavior that can be tested.

cd /path/to/git-root
omni run ~/.omnigent/agents/harness-engineering --harness codex

Use the planning skill for /absolute/path/to/project.
Build a notes app where a user can create a note, reload,
and still find it. Validate project_specs.xml.

Review the generated checks before continuing. Each check should describe behavior that QA can actually observe.

Created: project_specs.xml — the goal, design decisions, and stable Acceptance Checks.

Existing codebase: run /harness-setup Your goal instead. Setup inspects the current repository, runs the planner and the generator's initialization stage, preserves application files, and stops before implementation.
Monorepo: setup records independently runnable projects in .harness/projects.json. Select a project and run subsequent skills from its directory; each project keeps its own specification and execution queue while Git coordination remains repository-wide.

Monorepo path

Set up the root, code from a project

Run setup once at the Git root. It detects independently runnable or deployable projects, writes the routing registry, and asks which projects to initialize. Shared libraries are not separate projects unless they run or deploy independently.

cd /work/acme
/harness-setup Add customer account management

# Result
.harness/projects.json
apps/web/project_specs.xml
apps/web/feature_list.json
services/api/project_specs.xml
services/api/feature_list.json

Review each selected project's specification. Then start coding from the project that owns the observable outcome:

cd /work/acme/apps/web
/harness-generator

cd /work/acme/services/api
/harness-generator

The worker can edit shared packages or sibling services when the owning specification requires it. For a frontend/backend feature, keep the end-to-end Acceptance Check in one owning project—usually the frontend. Create a separate backend goal only when it is independently releasable.

Isolation: each project has its own specification, queue, journals, claims, Run State, and Goal Review. Git locks and integration remain repository-wide.

Step 6

Create the execution queue

Ask Omnigent to reconcile the specification before starting. This creates the queue and checks that every Acceptance Check is mapped.

Reconcile project_specs.xml for /absolute/path/to/project.
Confirm feature_list.json passes the generator check.

Created: feature_list.json, CONTEXT.md, harness-progress/, and Run State under .git/harness-runs/.

Step 7

Start or resume from Omnigent

Start from the Git root. Name the exact project directory, especially in a monorepo.

cd /path/to/git-root
omni run ~/.omnigent/agents/harness-engineering --harness codex

Act as the Control Host for /absolute/path/to/project.
Start or resume the harness. Use the project role file.
Continue until input is needed or run_completed is persisted.

The engine owns claims, retries, merges, Integrated Verification, and Goal Review. Omnigent is the control screen; it does not create another scheduler.

Created: per-context branches, worktrees, journals, Run State, routing evidence, and the existing integration checkpoints.

Step 8

Answer Input Requests from mobile

The Omnigent status and Input Request skills read durable state. A reply must target the blocked request and enter the existing explicit Resume path.

Show current status for /path/to/your-project.
For Input Request EVENT_ID, retry with this guidance: ...

Native operation remains available through /harness-generator and the detached control-host supervisor; Omnigent does not own a second scheduler.

Durability: unanswered requests remain in Run State and evidence even if the browser, phone, or Omnigent session disconnects.

Step 9

Verify the whole chain

CONTROL=~/.config/opencode/skills/harness-control-host/scripts/harness-control.mjs
1
The supervisor finishednode "$CONTROL" status --repo /path/to/project

Look for status: complete and supervisorPid: null.

2
The role config is validjq . .harness/roles.json
3
The plan is validnode ~/.config/opencode/skills/harness-generator/reconcile.mjs /path/to/your-project --check
4
All three checks passedjq 'all(.[]; .implementation and .qa and .integration)' /path/to/project/feature_list.json
5
Routing is recordedrg '^route=' .git/harness-runs/evidence
6
The goal actually passednode "$CONTROL" events --repo /path/to/project --consumer manual-check

Wait for a durable event whose kind is run_completed. An empty queue or an agent message is not completion.

Operations

Inspect, pause, resume, or stop

CONTROL=~/.config/opencode/skills/harness-control-host/scripts/harness-control.mjs
node "$CONTROL" status --repo /path/to/your-project
node "$CONTROL" capacity --repo /path/to/your-project --host opencode
node "$CONTROL" pause  --repo /path/to/your-project
node "$CONTROL" resume --repo /path/to/your-project
node "$CONTROL" stop   --repo /path/to/your-project

If a worker blocks, answer the exact Input Request in Omnigent or resume natively with explicit guidance. The harness preserves the branch, worktree, defect report, repair plan, and evidence.
