What the workflow provides

Not just agents. A system of checks and handoffs.

Each part has one job. Together they keep a large task understandable after hours, retries, parallel work, and new conversations.

01

Goal-first planning

The planner converts an idea into one observable Project Goal and stable Acceptance Checks: the proof needed before anyone can say “done.”

project_specs.xml
02

Dependency-aware queue

Checks become Work Items. Only work whose prerequisites are complete becomes ready, so independent work moves without ignoring real dependencies.

feature_list.json
03

Safe parallel work

Each context gets its own branch, folder, port, and Claim Lease. Multiple workers can move quickly without editing the same collision area.

isolated worktrees
04

Independent QA

The agent that writes the code does not grade its own work. QA checks behavior through a real browser or HTTP boundary and stores evidence.

expected vs observed
05

Planned repairs

A failure becomes a Defect Report. The orchestrator diagnoses it, writes a bounded Repair Plan, and gives that plan to the next coding attempt.

maximum 3 attempts
06

Proof after integration

Passing alone is not enough. Every Work Item is merged into the latest main, checked again, then the complete Project Goal receives a final review.

Goal Review
07

Durable memory

Machine state, a concise human journal, and separate evidence survive closed chats. A new session reads the files instead of guessing from memory.

resume, don't restart

Governed supervision

A singleton supervisor limits new workers using CPU, memory, system load, and provider quota. It can send progress and Input Requests through Telegram.

harness-control

Lesson 02

The completion contract

First define the outcome and its proof

The planner writes project_specs.xml. XML is simply labeled text: opening and closing labels tell people and programs what each paragraph means.

XMLproject_specs.xml
Question answered

What are we building, and what visible behavior proves it works?

Written by

The planner, with product decisions from the user.

Read by

The initializer, reconciler, workers, QA, and final Goal Review.

<project_goal>
  A visitor can publish a note and find it again after reloading.
</project_goal>

<acceptance_check
  id="AC-001"
  context="notes"
  category="functional"
  depends_on="">
  <description>
    A visitor publishes a note, reloads the page,
    and sees the same title and text.
  </description>
</acceptance_check>
id="AC-001"

A permanent name for this piece of proof. It should never be reused or silently changed.

context="notes"

The work area. Things likely to edit the same files belong in the same context.

depends_on=""

Which other checks must pass first. Empty means this work can start immediately.

Good check: “Publish a note, reload, and see the same content.” Weak check: “Notes are implemented.”
Lesson 03

Turning proof into work

The queue is a list of work—not the definition of success

reconcile.mjs makes sure every Acceptance Check has a matching Work Item in feature_list.json. JSON is a strict list of named values that programs can update safely.

{
  "id": "WI-AC-001",
  "context": "notes",
  "description": "A published note survives reload",
  "acceptance_checks": ["AC-001"],
  "depends_on": [],
  "implementation": false,
  "qa": false,
  "integration": false,
  "retries": 0
}
1implementation

The coding agent says its change is ready for inspection.

2qa

An independent agent observed the behavior in the isolated work area.

3integration

The same behavior still works after joining the latest shared product.

These values are progress markers. They do not replace the Project Goal. Even when every item says integration:true, the whole product must still pass Goal Review.

Lesson 04

Safe parallelism

Parallel work happens in separate lanes

A context is a collision boundary: a group of related Work Items likely to touch the same files. Each ready context receives a private Git branch and worktree—an isolated folder containing its version of the project.

Three contexts, one shared productMerges happen one at a time
flowchart TB
  Q["Dependency-aware queue"] --> A["accounts context<br/>worktree A : port 5170"]
  Q --> B["reports context<br/>worktree B : port 5171"]
  Q --> C["settings context<br/>worktree C : port 5172"]
  A --> L["Serialized merge lock"]
  B --> L
  C --> L
  L --> M["latest main"]
  M --> V["Integrated verification"]
  • Claim Lease: only one worker may own a context.
  • Dependency Graph: a context waits only for checks it truly needs.
  • Merge lock: parallel building is allowed; combining and checking the shared product is serialized.
  • Resource Governor: even ready work waits when CPU, memory, load, or provider quota has no safe slot.
More workers are not always faster. Useful parallelism is limited by ready contexts and real machine/provider capacity.
Lesson 05

The quality loop

Build, inspect, diagnose, then try again

One Attempt contains coding, isolated QA, and—if QA passes—Integrated Verification. A failed inspection does not go straight back to coding with a vague “fix it.”

The bounded repair loopMaximum three Attempts
flowchart TD
  C["Coding agent"] --> Q["Independent QA"]
  Q -->|"passes"| P["Checkpoint"]
  P --> I["Merge + Integrated Verification"]
  I -->|"passes"| D["Work Item complete"]
  Q -->|"fails"| R["Defect Report<br/>expected / observed / evidence"]
  I -->|"fails"| R
  R --> RP["Orchestrator writes Repair Plan"]
  RP --> N{"Attempt left?"}
  N -->|"yes"| C
  N -->|"no — 3 of 3"| B["Blocked Work Item<br/>ask the user"]
QA produces

Defect Report

Expected behavior, observed behavior, reproduction details, evidence, and affected Acceptance Checks.

Orchestrator produces

Repair Plan

Likely root cause, bounded actions for the next coding run, and how that fix must be validated.

After the third failed Attempt, the branch, worktree, state, journal, defects, plans, and evidence remain intact. The user receives an Input Request and can retry with guidance, pause, or stop.

Lesson 06

What “done” means

Completion is a system-level claim

A Work Item can pass alone and still break when combined with other work. That is why the workflow has two increasingly broad inspections after coding.

Local proof

Isolated QA

Does this behavior work in its private worktree?

Combined proof

Integrated Verification

Does it still work after merging into the latest main?

Whole-product proof

Goal Review

Do all checks and primary user journeys prove the Project Goal?

A Goal Review defect reopens the affected Work Items when their Acceptance Check IDs identify the repair area. Ambiguous defects, changed files during review, or exhausted Attempts block for user guidance. Only a persisted run_completed event with goal:true means complete.

Lesson 07

Surviving interruption

The conversation is disposable; the state is not

Closing a chat should not erase the work. The harness stores short machine and human records in the repository, then a new session reads them before claiming anything.

FileFormatWhat it remembers
project_specs.xmlXMLThe product goal, design decisions, and stable Acceptance Checks.
feature_list.jsonJSONThe Work Item queue, dependencies, three proof flags, and retry count.
.git/harness-runs/<context>.jsonJSONCurrent phase, owner, processes, heartbeat, Attempt, result, and next action.
harness-progress/<context>.mdMarkdownA concise journal people can read: transitions, defects, plans, and evidence links.
.git/harness-runs/evidence/ArtifactsDetailed browser, HTTP, command, and runtime output kept outside the journal.
.git/generator-claims.jsonJSONWhich contexts, branches, worktrees, and ports are currently claimed.

On restart, live process IDs and fresh heartbeats are left alone. Dead local work resumes in the same worktree. A stale claim from another machine asks the user before takeover. Blocked work never resumes without explicit guidance.

Lesson 08

Running for hours or days

A Control Host watches; the harness decides

Hermes, nanobot, pi, or another long-lived agent can accept your goal and talk to Telegram. It does not invent its own scheduling rules. It uses the harness control interface.

Control and notification pathMessages are durable before delivery
sequenceDiagram
  participant U as User on Telegram
  participant H as Control Host
  participant S as Harness Supervisor
  participant W as Governed Workers
  H->>S: start / status / events
  S->>W: admit work when capacity allows
  W-->>S: durable Run State + result
  S-->>H: 15-minute progress event
  H-->>U: progress summary
  S-->>H: immediate Input Request #42
  H-->>U: reason, evidence, choices
  U->>H: retry #42 with guidance
  H->>S: idempotent response for #42
  S->>W: retry when capacity allows
Control filePurpose
.git/harness-control/state.jsonSupervisor status, capacity, workers, progress, and pending Input Requests.
supervisor.lock/A singleton lease so two supervisors cannot each spend the full quota.
events.jsonlOne JSON object per important event, in order. This is not a chat transcript.
responses/<id>.jsonOne durable, idempotent user decision for an Input Request.
cursors/<consumer>.jsonThe last event successfully delivered to each channel, such as Telegram.

Events are acknowledged only after delivery. If Telegram fails, the cursor does not move, so the message returns on the next poll. A context-level blocker does not stop unrelated ready work. Goal-wide pause is reserved for problems that make all useful work unsafe or impossible.

Lesson 10

Know the edges

What this workflow does not promise

It cannot rescue a vague goal

Automation can repeatedly satisfy the wrong requirement. Review the Project Goal and Acceptance Checks before long execution.

Flags are not proof by themselves

A bad QA agent can still mark a flag. Real browser/HTTP evidence and Goal Review reduce this risk; they do not make testing infallible.

Provider quotas are not discoverable everywhere

The supervisor uses a configured concurrent slot count and reacts to rate-limit errors. It cannot predict undocumented rolling limits.

Parallelism has a collision boundary

Work inside one context is sequential. Poor context design either causes file conflicts or removes useful parallelism.

Cross-machine recovery needs shared state

Another machine must see the same Git state and reliable clocks. Fresh remote leases are intentionally not stolen.

Native Windows control is limited

The supervisor invokes the Bash claim helper. Use Git Bash or WSL for the complete long-running control path.

It does not store secrets for you

Credentials belong in host configuration or environment variables. Backups redact secrets and prompt again during restore.

GitHub Pages is documentation only

This website explains the harness. The workflow itself runs on your machine or infrastructure with authenticated agent CLIs.

Put it into practice

The shortest path from idea to run

  1. 1
    Install the harness

    Run the installer and choose one available worker host.

  2. 2
    Plan the product

    Run /harness:planner. Read the generated goal and checks.

  3. 3
    Start building

    Run /harness:generator, or start the supervisor after initialization.

  4. 4
    Watch durable status

    Use the status/events commands; respond by Input Request ID when needed.

  5. 5
    Accept only Goal Review

    Wait for the persisted run_completed event.
